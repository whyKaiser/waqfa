## الفكرة

التطبيقات المالية التقليدية تشرح للمستخدم أين صرف بعد وقوع الأثر. **وقفة** يتدخل قبل قرار الشراء: يحاكي أثر قسط جديد على التدفق النقدي، يوضح أسباب الخطر، يختبر القدرة على تحمل الطوارئ، ويقترح بديلًا أكثر أمانًا.

تموضع المنتج المستهدف هو **جدار حماية للقرار المالي قابل للدمج داخل تطبيق مصرف الإنماء**. النموذج الحالي تطبيق أولي مستقل، وليس متصلًا فعليًا بأنظمة الإنماء أو أي بنك أو مزود BNPL.

وقفة ليس تقييمًا ائتمانيًا رسميًا ولا يتخذ قرار منح التمويل. مؤشره الحالي أداة وقائية وتجريبية لدعم قرار المستخدم.

## حالة المشروع والادعاءات

### مبني الآن

- تطبيق Flutter عربي يعمل على Android وiOS والويب.
- إدخال الدخل والمصروفات والأقساط وتحليلها.
- تحليل توليدي عربي عبر Groq مع fallback محلي.
- محرك قرار مالي حتمي وقابل للتفسير.
- مقارنة الخطر قبل القسط الجديد وبعده.
- محاكاة يومية لمدة 90 يومًا و72 مسارًا احتماليًا قابلًا لإعادة التشغيل.
- **موعد السقوط المالي:** أول بقاء متوقع تحت احتياطي الأمان ثلاثة أيام متتالية، مع احتمال وافتراضات معلنة.
- **أقل تدخل منقذ:** بحث حتمي بين خفض القسط والتأجيل وحماية جزء من الصرف المتغير.
- مقارنة زمن التعافي والأيام الهشة قبل القرار وبعده وبعد التدخل.
- **خطة وقفة:** هدف 7/30/90 يومًا بخطة أسبوعية وأرقام من محرك محلي، لا من LLM.
- **سجل المستقبل البديل:** أثر مضاد للواقع محفوظ محليًا دون تسميته ادخارًا فعليًا.
- عوامل الخطر وثلاثة اختبارات صدمة.
- متعلم سلوكي محلي من نوع UCB1 يختار بين ثلاثة تدخلات ويحدث عدادات مجهولة بعد قرار المستخدم.
- مختبر تحقق داخل التطبيق يعرض المقاييس والحدود بوضوح.
- بدائل قابلة للتجربة مثل خفض القسط أو مهلة 24 ساعة.
- قراءة بيانات الفاتورة عبر نموذج رؤية.
- تخزين الملف والسجل محليًا على الجهاز.
- شاشة **محاكاة** للمصرفية المفتوحة؛ لا يوجد ربط مصرفي حي.

### متحقق اصطناعيًا

- توجد 100 حالة اصطناعية موزعة على خمسة أنماط، تولد بـseed ثابت وتُحاكى يوميًا لمدة 90 يومًا.
- نتائج التشغيل الحالي: اكتشاف `47/47` حالة حرجة اصطناعية، دون حالة فائتة، مع `28` إنذارًا احترازيًا زائدًا من أصل 53 حالة غير حرجة (`52.83%`).
- وسيط الإنذار المبكر 6 أيام. وعند تنصيف القسط انخفض المؤشر `13.11` نقطة في المتوسط للحالات المنبّهة، وتجنبت المحاكاة نتيجة حرجة في `17.02%` من الحالات الحرجة.
- الشخصيات التجريبية مثل «سارة» **ليست مستخدمين حقيقيين**، وبياناتها ليست بيانات بنك.
- هذه الاختبارات تثبت سلوك النموذج حسب قواعده، ولا تثبت دقة واقعية أو خفضًا في التعثر.
- لا يعتمد المشروع أي نسبة دقة أو أثر ما لم تُحسب من تشغيل موثق.

### مرحلة Pilot

- دمج المحرك داخل قناة مصرفية بموافقة العميل.
- استبدال محاكاة الربط بتكامل فعلي عبر قناة أو مزود مرخص.
- معايرة نماذج التنبؤ على بيانات مصرح بها وقياس الإنذارات الكاذبة والانحياز.
- مقارنة تغير القرار والأثر المالي الفعلي مع تجربة اعتيادية.

## رحلة «وقفة قبل تدفع»

1. يدخل المستخدم وضعه المالي وقرار الشراء المقترح.
2. يحسب المحرك مؤشر القرار قبل القسط وبعده.
3. يعرض توقع التدفق النقدي خلال 90 يومًا.
4. يفسر أهم العوامل التي رفعت الخطر.
5. يختبر القرار أمام فاتورة مفاجئة، وصيانة سيارة، وتأخر جزء من الدخل.
6. يبحث عن أقل تدخل، يقارن التعافي، ثم يحفظ أثر القرار محليًا عند اعتماده.

### فصل الحساب عن الذكاء التوليدي

لا يُطلب من النموذج اللغوي اختراع الحسابات المالية. `FinancialDecisionEngine` و`FinancialPreventionEngine` و`GoalPlanEngine` تحسب الأرقام بشكل قابل للاختبار، بينما يستخدم الذكاء الاصطناعي لفهم السياق وقراءة الفاتورة وصياغة شرح عربي دون تغيير أرقام الخطة.

نماذج التنبؤ المتقدمة ومعايرة احتمال الخطر على بيانات واقعية جزء من مرحلة الـPilot، وليست ادعاءً عن النموذج الحالي.

## الخصوصية والأمان

- الملف والسجل محفوظان محليًا في النموذج الحالي.
- الحسابات الأساسية لمحرك القرار تعمل محليًا.
- عند استخدام خصائص Groq، تُرسل المدخلات اللازمة إلى API خارجي؛ لذلك لا يدّعي المشروع أن كل المعالجة محلية.
- مفتاح Groq يبقى خارج Git ويُمرر للنموذج الأولي عبر `GROQ_API_KEY` و`--dart-define` أثناء البناء.
- تضمين المفتاح في تطبيق العميل مناسب للعرض التجريبي فقط؛ قبل الإنتاج يجب نقله إلى backend proxy مع ضبط الصلاحيات والحصص.
- أي تكامل مصرفي مستقبلي يحتاج موافقة صريحة، وتقليل البيانات، وتشفيرًا، وصلاحيات واضحة، ومسارًا مرخصًا.

## نموذج العمل المستهدف: B2B2C

- **المستخدم:** يحصل على وقفة مجانًا داخل قناة البنك.
- **العميل الدافع:** مصرف الإنماء أو الجهة المالية المشغلة.
- **الإيراد:** رسوم إعداد وتكامل، ثم ترخيص سنوي مؤسسي وشريحة استخدام بحسب العملاء النشطين أو القرارات المحللة.
- **القيمة المقترحة للبنك:** تحسين الصحة المالية والثقة والتفاعل، واختبار فرضية خفض الضغط المالي والتأخر القابل للتجنب.

لا توجد حاليًا أسعار معتمدة أو هوامش ربح مثبتة. كما لا يدّعي المشروع أنه خفّض التعثر؛ هذا أثر يحتاج إلى Pilot مصرفي ومجموعة مقارنة.

## Pilot مقترح

1. تشغيل محكوم مع عملاء موافقين داخل بيئة الإنماء.
2. قياس القرارات التي أُجلت أو خُفضت أو أُلغيت بعد التنبيه.
3. قياس تغير هامش الرصيد الآمن والإنذارات الكاذبة ورضا المستخدم.
4. قياس التأخر الفعلي لاحقًا مقابل تجربة اعتيادية إذا توفرت البيانات والمدة والموافقات.
5. مراقبة الدقة والانحياز والانحراف قبل أي توسع.

## التشغيل

```bash
flutter pub get

# يعمل المحرك المالي محليًا دون مفتاح. لتفعيل خصائص Groq:
flutter run --dart-define=GROQ_API_KEY="<YOUR_GROQ_KEY>"
```

في Codemagic، يُحفظ `GROQ_API_KEY` كمتغير Secret داخل مجموعة `waqfa_secrets` التي يستوردها `codemagic.yaml`.
