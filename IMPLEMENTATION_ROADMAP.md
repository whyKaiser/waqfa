# خارطة تنفيذ «وقفة»

> آخر تحديث: 16 يوليو 2026
> الأولوية: Demo مستقر ومترابط أولًا، ثم تحسينات الـPilot.
> مقياس التعقيد: S أقل من نصف يوم، M نصف–يومين، L ثلاثة–خمسة أيام، XL أكثر من أسبوع لفريق صغير.

## قواعد التنفيذ

- لا إعادة كتابة شاملة.
- لا تغيير خوارزمية مالية دون اختبار قبل/بعد.
- كل رقم مالي من محرك، لا من نص AI.
- المسار الأساسي يعمل دون API.
- بيانات Demo ثابتة وموسومة «اصطناعية».
- لا تخزين أسرار في Git أو اعتبار `--dart-define` حماية إنتاجية.
- كل مرحلة قابلة للتسليم والرجوع دون كسر الشاشات السابقة.

## Phase 0 — تثبيت Baseline والاختبارات

### الهدف

توثيق الحالة الفعلية قبل تغيير المحركات، ومنع نسبة أي خلل قديم للميزات الجديدة.

### الحالة

المستودع يحتوي اختبارات وbenchmark ونتائج موثقة، لكن يجب إعادة تشغيل الأوامر بعد اكتمال كل تغييرات الفرع؛ لا يكفي وجود الملفات.

### الملفات المتأثرة

- `pubspec.yaml`
- `README.md`
- `docs/MODEL_CARD.md`
- `test/**`
- `tool/risk_benchmark.dart`
- `tool/risk_threshold_sweep.dart`
- `docs/benchmark_results.json`
- ملفات CI مثل `codemagic.yaml`

### الاعتمادات

- Flutter SDK المتوافق مع `pubspec.yaml` وCodemagic.
- Dart SDK.
- لا حاجة لمفتاح Groq لاختبارات المسار المحلي.

### الأعمال

1. حفظ `git status` والفرع والـcommit.
2. تشغيل `flutter pub get`.
3. تشغيل `flutter analyze` و`flutter test`.
4. تشغيل benchmark وthreshold sweep بالبذرة الموثقة.
5. مقارنة README وModel Card بنتائج التشغيل.
6. تثبيت سيناريو Demo واحد باختبار، وعدم تغيير أرقامه أثناء العرض.
7. توثيق أي ملف مستخدم غير مرتبط وعدم لمسه.

### المخاطر

- اختلاف Flutter محليًا عن CI.
- نتائج benchmark قد تتغير إذا تغيرت العتبات أو تعريف الحالة الحرجة.
- وثائق قديمة تحتوي أرقامًا مختلفة.

### الاختبارات

- كل الاختبارات الحالية.
- إعادة تشغيل benchmark مرتين والتأكد من تطابق الناتج.
- فحص عدم وجود مفتاح فعلي عبر `rg` لأنماط الأسرار.

### معايير القبول

- `flutter analyze` بلا أخطاء.
- `flutter test` ناجح بالكامل.
- benchmark قابل لإعادة التشغيل ويذكر أنه اصطناعي.
- نسخة المحرك والعتبات والseed موثقة.
- لا ادعاء نجاح بناء iOS/Android إلا مع سجل CI فعلي.

### التعقيد

**S–M** حسب الأعطال المكتشفة.

## Phase 1 — المحرك الأساسي

### الهدف

بناء نتيجة وقاية موحدة: توأم مبسط، موعد ضغط، أقل تدخل، وزمن تعافٍ.

### الملفات المتأثرة

- `lib/services/financial_decision_engine.dart`
- `lib/services/cash_flow_engine.dart`
- `lib/services/financial_prevention_engine.dart` أو Facade مكافئ
- `lib/services/intervention_search_engine.dart` إن فُصل البحث
- `lib/models/**` إن أُنشئ مجلد نماذج
- `test/financial_decision_engine_test.dart`
- `test/financial_prevention_engine_test.dart`
- `test/cash_flow_benchmark_test.dart`
- `lib/services/risk_benchmark.dart`

### الاعتمادات

- Phase 0 ناجحة.
- تعريف مكتوب لـ«نقطة الضغط» و«التعافي».
- نطاق بحث تدخل محدود ومعلن.

### الأعمال

1. إنشاء `FinancialTwinInput` و`PreventionAnalysis` غير قابلين للتعديل.
2. توحيد التحقق من القيم والحدود.
3. تشغيل baseline وبعد القرار عبر `DailyCashFlowEngine`.
4. تحويل day index إلى تاريخ باستخدام `startDate` فقط.
5. حساب موعد الضغط أو نتيجة «لا ضغط ضمن الأفق».
6. تعريف وحساب زمن التعافي قبل/بعد.
7. توليد مرشحي التدخل وإعادة محاكاة كل مرشح.
8. اختيار أصغر تدخل قابل للتنفيذ بدالة هدف موثقة.
9. جعل benchmark يستخدم نفس مدخل/ناتج المحرك المعروض.

### المخاطر

- تعريف «تعافٍ» غير مستقر يغير الأرقام دون ملاحظة.
- مبلغ الحماية قد يُعامل خطأً كدخل.
- التأجيل خارج الأفق قد يبدو حلًا وهو ترحيل للمشكلة.
- اختلاف الدرجة عن حقيقة التدفق اليومية.

### الاختبارات

- لا نقطة ضغط، نقطة في اليوم الأول، ونقطة في اليوم 89.
- دخل والتزام في اليوم نفسه مع ترتيب ثابت.
- قيم صفرية، سالبة، `NaN`, `Infinity`، وأرقام ضخمة.
- مستخدم مستقر، BNPL متزامن، دخل متذبذب اصطناعي، هش للطوارئ، وتوقيت سيئ.
- تدخل بسيط ناجح، تدخل مركب، ولا تدخل متاح.
- تعافي فوري، بعد راتب، وغير متحقق خلال الأفق.
- Snapshot لأرقام سيناريو Demo.

### معايير القبول

- نتيجة واحدة تستخدمها الشاشات والاختبارات.
- لا حساب مالي داخل Widget.
- كل تدخل أعيد اختباره فعليًا بالمحرك.
- لا يعرض التاريخ إذا لم توجد نقطة ضغط.
- لا يعرض احتمالًا غير معاير.
- نفس المدخلات تعطي نفس النتيجة.

### التعقيد

**L**.

## Phase 2 — تجربة القرار: حارس الشراء

### الهدف

تحويل المحرك إلى رحلة يمكن فهمها خلال ثوانٍ وشرحها خلال 60–75 ثانية.

### الملفات المتأثرة

- `lib/screens/input_screen.dart`
- `lib/screens/result_screen.dart`
- `lib/screens/simulator_screen.dart`
- `lib/screens/receipt_scan_screen.dart`
- `lib/screens/home_screen.dart`
- مكونات UI مشتركة مقترحة داخل `lib/widgets/**`
- `test/widget_test.dart`
- اختبارات Widget جديدة للحارس

### الاعتمادات

- Phase 1 مستقرة.
- Design System موثق في `DESIGN_SYSTEM_AMAD.md`.
- بيانات Demo ثابتة.

### الأعمال

1. زر رئيسي واضح «اختبر قرارك قبل تدفع».
2. تحقق فوري من المدخلات مع رسائل عربية محددة.
3. ترتيب النتيجة: ماذا، متى، لماذا، التدخل، الأثر، التالي.
4. عرض أهم سببين بدل ازدحام كل العوامل.
5. مقارنة قبل/بعد للتدخل نفسه: درجة، أيام هشاشة، تعافٍ.
6. إظهار الافتراضات في تفاصيل قابلة للفتح.
7. إبقاء تصوير الفاتورة إدخالًا اختياريًا خارج المسار الأساسي.
8. حالات loading/error/offline واضحة.

### المخاطر

- ازدحام الشاشة على iPad أو الهاتف.
- ألوان الخطر قد تبدو مخيفة أو عقابية.
- عرض يوم حرج دون شرح الافتراضات يفهم كتنبؤ مؤكد.
- فشل AI أو قراءة الفاتورة أثناء Demo.

### الاختبارات

- تدفق يدوي كامل دون مفتاح AI.
- سيناريو آمن، يحتاج وقفة، وخطر.
- غياب التاريخ الحرج.
- «لا يوجد تدخل منقذ ضمن الخيارات».
- RTL وتكبير النص 200% دون clipping أساسي.
- رفض الموافقة السحابية وعدم إرسال الصورة.

### معايير القبول

- المستخدم يفهم النتيجة الأساسية خلال 10 ثوانٍ.
- الـDemo الكامل لا يزيد عن 75 ثانية.
- لا شاشة أساسية تعتمد على شبكة.
- الفرق بين «محاكاة» و«توقع مصرفي» ظاهر.
- كل زر ينتج خطوة واضحة ولا توجد مسارات ميتة.

### التعقيد

**M–L**.

## Phase 3 — خطة وقفة والإنقاذ

### الهدف

تحويل التدخل المحسوب إلى خطة 7/30/90 يومًا مرتبطة بهدف المستخدم.

### الملفات المتأثرة

- `lib/services/waqfa_plan_engine.dart`
- `lib/services/ai_service.dart` أو واجهة Provider جديدة
- `lib/screens/waqfa_plan_screen.dart`
- `lib/screens/home_screen.dart`
- `lib/services/profile_service.dart`
- `lib/services/plan_repository.dart` أو التخزين المكافئ
- اختبارات خطة وStructured Output

### الاعتمادات

- Phase 1 و2.
- أنواع أهداف محددة.
- تعريف مبلغ قابل للحماية وحد صرف آمن.

### الأعمال

1. اختيار هدف: طوارئ، تخفيف أقساط، شراء، سفر، زواج، سيارة، مبلغ، آخر.
2. إدخال المبلغ والمدة عندما يلزم.
3. اختبار إمكانية الهدف بالمحرك.
4. عند عدم الإمكان، تشغيل أقل تدخل لجعل الخطة أقرب أو التصريح بعدم كفاية التعديل.
5. توليد خطوات أسبوعية محلية بأرقام المحرك.
6. إنشاء واجهة AI provider-agnostic وStructured JSON.
7. التحقق أن AI لم يضف رقمًا غير موجود في payload.
8. fallback محلي كامل عند غياب الشبكة أو المفتاح.

### المخاطر

- وعد هدف غير ممكن بسبب أفق 90 يومًا محدود.
- AI يعيد صياغة رقم بشكل خاطئ.
- خطة عامة كثيرة النصوص.
- تخزين هدف حساس دون تشفير.

### الاختبارات

- هدف ممكن، ممكن بتدخل، وغير ممكن ضمن الأفق.
- هدف بلا مبلغ، مدة غير صحيحة، ومبلغ غير منطقي.
- مجموع الخطوات لا يتجاوز الفائض.
- Structured JSON صحيح/ناقص/تالف/يحمل رقمًا مخترعًا.
- timeout و401 و429 يعيدان fallback.

### معايير القبول

- جميع الأرقام قابلة للإشارة إلى حقل محسوب.
- الخطة تعرض 3–5 خطوات، لا نصًا عامًا طويلًا.
- تغيير الهدف يعيد الحساب.
- لا وصف «مستشار مالي مرخص» ولا توصية استثمار.
- تعمل الخطة دون API.

### التعقيد

**L**.

### الاعتمادات

- معرف قرار ثابت.
- ناتج قبل/بعد من Phase 1.
- قرار المستخدم الصريح.

### الأعمال

1. تسجيل القرار المعدل/المؤجل/الملغى/المستمر.
2. حفظ أثر متوقع: تغير الخطر، الأيام الهشة، والتعافي.
3. فصل التزام لم يُنشأ عن مبلغ ادخار فعلي.
4. منع التكرار.
5. عرض أن بيانات الـDemo اصطناعية.
6. إبقاء UCB1 الحالي محدودًا باختيار أسلوب الرسالة، لا الادعاء بتعلم بصمة مالية.
7. إضافة مسح السجل وتعطيل التعلم.

### المخاطر

- double counting.
- خلط الأثر المتوقع بالواقع.
- استنتاج شخصي من عينة صغيرة.
- فقد السجل عند تبديل الجهاز في Prototype.

### الاختبارات

- تكرار `decisionId`.
- قرار مستمر بلا أثر متجنب.
- خفض قسط مع حساب الفرق الصحيح.
- تأكيد تحويل فعلي منفصل.
- أقل من الحد الأدنى للتعلم لا يولد «بصمة».
- مسح البيانات يزيل السجل والتعلم.

### معايير القبول

- لا يظهر لفظ «ادخرت» دون تأكيد تحويل فعلي.
- كل أثر يحمل `calculationVersion` و`syntheticDemo` عند اللزوم.
- لا تظهر بصمة شخصية زائفة.
- سجل واضح وقابل للمسح.

### التعقيد

## Phase 6 — رؤية التكامل المصرفي

### الهدف

تحويل النموذج إلى Pilot منضبط داخل قناة مصرف الإنماء، دون تمثيل الرؤية كميزة مبنية الآن.

### الملفات المتأثرة

- وثائق API وData Contract
- Threat model وPrivacy impact assessment
- Backend جديد خارج تطبيق Flutter
- Repository وAI provider production
- Feature flags وtelemetry
- بروتوكول Pilot ولوحة قياس

### الاعتمادات

- موافقة جهة مصرفية وفرق الأمن والخصوصية والامتثال.
- قناة بيانات مرخصة.
- تعريف شرائح Pilot ومجموعة مقارنة.
- اتفاق retention وحذف ومراجعة بشرية.

### الأعمال

1. backend proxy لخدمات AI.
2. تكامل read-only للمعاملات والدخل والالتزامات بعد الموافقة.
3. كشف recurring والـBNPL والتواريخ مع مراجعة المستخدم.
4. معايرة العتبات على بيانات Pilot مصرح بها.
5. وسادة وقفة كاقتراح فقط أولًا؛ أي حماية مبلغ فعلية تحتاج موافقة مستقلة.
6. مراقبة false alerts والانحياز والانحراف.
7. kill switch وfallback ومراجعة حوادث.
8. تجربة مقارنة لقياس تغير القرار قبل قياس التأخر طويل الأجل.

### المخاطر

- تصنيف خاطئ للمعاملات أو دخل متذبذب.
- استخدام البيانات لغرض خارج الموافقة.
- إنذار زائد يسبب فقدان الثقة.
- تحول الأداة الوقائية إلى قرار ائتماني غير مقصود.
- مسؤولية تنفيذ حماية أموال.

### الاختبارات

- Contract tests وsandbox مصرفي.
- اختبارات صلاحيات وسحب موافقة وحذف.
- penetration test وsecret rotation.
- انقطاع المزود والـkill switch.
- عدالة الأداء حسب الشرائح المسموح بقياسها.
- تجربة مستخدم مع لغة غير مخيفة.

### معايير القبول

- موافقة صريحة وقابلة للسحب.
- لا سر داخل العميل.
- جميع الأفعال المالية قابلة للتدقيق والتراجع حيث ينطبق.
- مقاييس Pilot متفق عليها قبل الإطلاق.
- لا توسع قبل بلوغ حدود false alerts والسلامة.

### التعقيد

**XL**؛ مشروع مؤسسي متعدد الفرق.

## ترتيب التنفيذ الحرج للهاكاثون

1. Phase 0 كاملة.
2. من Phase 1: Facade، موعد الضغط، التدخل، التعافي، واختبارات سيناريو Demo.
3. من Phase 2: شاشة واحدة قوية للحارس.
4. من Phase 3: خطة محلية قصيرة؛ AI تحسين اختياري.
5. من Phase 4: تسجيل قرار واحد وأثره المتوقع.
6. من Phase 5: polish للشاشات الداخلة في العرض فقط ثم مواد التحكيم.

أي وقت زائد يذهب لإصلاح الاستقرار والوصول، لا لإضافة ميزة ثامنة.

- Unit tests ناجحة.

- analyze/test/build ناجحة ومسجلة.
- لا أسرار مكشوفة في Git.
- README/Model Card/عرض متطابقة.
- جميع ميزات Pilot موسومة بوضوح.
